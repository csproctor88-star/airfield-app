# Glidepath -- NotebookLM Prompt Library

## What This Document Is

This file contains structured prompts designed for Google NotebookLM. Each prompt is self-contained and can be pasted directly into NotebookLM to generate one of three artifact types:

1. **Video Overviews** -- AI-generated audio/video briefings for different audiences
2. **Infographics** -- Visual one-pagers and data-driven graphics
3. **Slide Decks** -- Presentation-ready briefing materials

### NotebookLM Setup

**Notebook Organization — Use One Master Notebook:**
Create a single notebook called **"Glidepath"** and load all source documents into it. Since every prompt references the same brand guide, statistics, and module inventory, one notebook ensures consistent context across all artifact types. You do not need separate notebooks per section.

**Custom Notebook Settings:**
No custom notebook-level settings are required. Each prompt below is self-contained with all necessary style, tone, and content guidance. NotebookLM will use both the uploaded sources and the prompt instructions to generate output.

**Source Documents — Upload All of These:**

- `docs/GLIDEPATH_CAPABILITIES_BRIEF.md`
- `docs/Glidepath_AFWERX_Spark_Proposal.md`
- `docs/GLIDEPATH_BETA_TESTER_OVERVIEW.md`
- `docs/Glidepath_SRS_v3.0.md`
- This file itself (for the Brand & Style Guide section)

### Where Each Prompt Goes

| Section | Artifact Type | Where to Paste the Prompt |
|---------|--------------|--------------------------|
| Section 1 | Video Overview | **Audio Overview** — Click "Generate" under Audio Overview, select "Customize," and paste the prompt into the custom instructions field. NotebookLM generates an audio briefing from your sources and the prompt guidance. |
| Section 2 | Infographic | **Notebook Chat** — Paste the prompt directly into the chat input. NotebookLM generates a structured content outline matching the layout and data specs. Use the output as a content brief for your design tool (Canva, Figma, etc.). |
| Section 3 | Slide Deck | **Notebook Chat** — Paste the prompt directly into the chat input. NotebookLM generates slide-by-slide content with titles, bullets, and presenter notes. Use the output to build the deck in Google Slides, PowerPoint, or your preferred tool. |

### 1C. Technical / Developer Overview

**Target audience**: Platform One engineers, software developers, DevSecOps teams, AFWERX technical evaluators
**Purpose**: Communicate the architecture, stack, and deployment readiness
**Tone**: Engineer-to-engineer -- precise, technically fluent, no hand-waving

```
Create a video overview artifact (3-4 minutes when read aloud) about
the technical architecture and development approach behind Glidepath.

TARGET AUDIENCE: Platform One engineers, DoD software factory teams, DevSecOps
personnel, AFWERX technical evaluators, and any developer assessing this
application for enterprise deployment.

OPENING -- WHAT THIS IS (15 seconds):
Glidepath is a production-grade Progressive Web Application for USAF Airfield
Management operations. Version 2.9.0. 15 modules, 48 routes, 130+ source files,
25+ database tables, 56 migrations, zero TypeScript errors. Built in 21 days
using AI-assisted development with Claude Code.

STACK OVERVIEW (45 seconds):
The technology choices were made for three reasons: Platform One compatibility,
zero vendor lock-in, and operational reliability.

- Framework: Next.js 14.2 with the App Router. Server-side rendering where it
  matters, client components for interactive modules.
- Language: TypeScript 5.9 with strict mode. Zero type errors across the entire
  codebase.
- Database: Supabase backed by PostgreSQL. 25+ tables, 56 schema migrations.
  Supabase provides auth, real-time subscriptions, storage, and the PostgREST
  API layer.
- Maps: Mapbox GL JS for airfield visualization, satellite imagery, and
  geospatial features including geodesic obstruction evaluation math.
- PDF Generation: jsPDF with jspdf-autotable for structured report output
  including embedded photos and Mapbox satellite map thumbnails.
- Excel Export: SheetJS and exceljs for styled spreadsheet output.
- Styling: Tailwind CSS with a custom dark-theme design system. CSS custom
  properties for the color palette: cyan primary, dark navy background, dark
  slate surfaces.
- PWA: Service worker for offline capability, installable on mobile and desktop.

ARCHITECTURE PATTERNS (45 seconds):
The codebase follows consistent patterns across all 15 modules:

- CRUD operations live in `lib/supabase/<entity>.ts` files. Each file creates a
  Supabase client, performs a null check, and falls back to demo data arrays when
  Supabase is not configured. This means the app runs in full demo mode with zero
  backend setup.
- Page structure follows Next.js App Router conventions: list pages at
  `app/(app)/<module>/page.tsx`, create pages at `new/page.tsx`, detail pages at
  `[id]/page.tsx`.
- Shared UI components in `components/ui/`: Badge, StatusBadge, SeverityBadge,
  ActionButton -- all styled to the dark-theme design system.
- Type safety through `lib/supabase/types.ts` with Database type and convenience
  aliases.
- Constants and configuration in `lib/constants.ts` -- type configs, status
  configs with color, background color, and label mappings.
- Activity logging via `logActivity()` for audit trail persistence.
- Installation context via `useInstallation()` hook providing installationId,
  areas, userRole, and CE shop data.

DATABASE DESIGN (30 seconds):
PostgreSQL with 25+ tables. Key design decisions:
- Entity-specific foreign keys for the photos table (discrepancy_id, check_id,
  inspection_id) rather than generic entity_id/entity_type polymorphism.
- Built-in installation directory covering 155 US military bases.
- 1,000+ aircraft records with ACN/PCN data for pavement analysis.
- 70 regulation references with metadata for the Regulations Library.
- RLS is disabled for MVP -- this is documented tech debt for production hardening.

DEPLOYMENT READINESS (30 seconds):
The application is ready for containerization and CI/CD pipeline integration.
The stack aligns with Platform One's preferred technologies. Key deployment
considerations:
- Environment variables for Supabase URL, anon key, and Mapbox token.
- Demo mode runs with zero configuration for evaluation.
- Storage uses Supabase Storage with explicit RLS policies on the photos bucket.
- The open-source stack (Next.js, PostgreSQL, no proprietary dependencies) means
  no licensing barriers for DoD deployment.

DEVELOPMENT APPROACH (20 seconds):

with no formal software engineering background -- using Claude Code for

CLOSE (15 seconds):
```
